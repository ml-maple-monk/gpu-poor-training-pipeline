# Verda Local GPU Serverless Container Emulator

Run a local Docker environment that mirrors Verda (formerly DataCrunch) Cloud GPU Serverless Containers for pre-deploy debugging with production-fidelity runtime.

## Quickstart (5 commands)

```bash
# 1. Parse credentials from ./secrets
./scripts/parse-secrets.sh

# 2. Run preflight checks (WSL2 + GPU + file permissions)
make preflight

# 3. Build image (tagged with git short SHA)
make build

# 4. Start the container
make run

# 5. Verify health
curl http://localhost:8000/health
```

## WSL2 Prerequisites

- **Windows NVIDIA driver** ≥ 527.x with WSL2 CUDA support
- **nvidia-container-toolkit** installed in WSL2: `sudo apt install nvidia-container-toolkit`
- **Docker with GPU support**: `docker info | grep nvidia` must show `nvidia` runtime
- **Project on ext4**: must NOT be under `/mnt/c` — use `~/workspace/` or similar
- **systemd** (recommended): add `[boot]\nsystemd=true` to `/etc/wsl.conf`

Verify GPU access:
```bash
nvidia-smi
docker run --rm --gpus all nvidia/cuda:12.4.1-runtime-ubuntu22.04 nvidia-smi
```

## Secret Files

| File | Purpose | Scope |
|---|---|---|
| `.env.inference` | `VERDA_INFERENCE_TOKEN` — bearer token for `/infer` | Container-only (loaded via `env_file`) |
| `.env.mgmt` | `VERDA_CLIENT_ID` + `VERDA_CLIENT_SECRET` — Verda management API auth | Host-only (never passed to container) |

Both files are generated by `./scripts/parse-secrets.sh` from the plaintext `./secrets` file and set to mode 600.

The inference token in `.env.inference` is a **local-dev-only random value** — not a Verda-issued token. Rotate it by deleting `.env.inference` and re-running `parse-secrets.sh`.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `VERDA_REQUIRE_GPU` | `1` (set by compose) | If `1`, container requires GPU; `/health` returns 503 without GPU unless `ALLOW_DEGRADED=1` |
| `ALLOW_DEGRADED` | unset | Set to `1` to allow CPU-only operation; `/health` returns 200 with `X-GPU-Fidelity: degraded` |
| `WAIT_DATA_TIMEOUT` | `30` | Seconds to wait for `/data` to be writable before fatal exit |
| `APP_PORT` | `8000` | Port exposed by the container |
| `VERDA_PULL_MODEL` | unset | If set, logged as a model prefetch stub (extend `start.sh` for real pull logic) |

## Compose Variants

### `make run` — GPU mode (default)
Uses `docker-compose.yml`. Requires `nvidia-container-toolkit` and a CUDA-capable GPU.

### `make cpu` — CPU degraded mode
```bash
make cpu
```
Applies `docker-compose.cpu.yml` overlay: removes GPU reservation, sets `ALLOW_DEGRADED=1`.
Use for local development without a GPU or CI environments.

### `make nvcr` — NGC PyTorch base
```bash
make nvcr
```
Applies `docker-compose.nvcr.yml` overlay: swaps base image to `nvcr.io/nvidia/pytorch:24.08-py3`.
Use when your workload requires the NGC-bundled PyTorch + CUDA toolkit (~8GB vs ~3GB for bare CUDA).

## Security Follow-up

**Accepted tech debt:** The `./secrets` file is stored as plaintext on disk. This is acceptable for local development but should be addressed before team or CI use:

- **[sops](https://github.com/mozilla/sops)** — encrypts secrets files with AWS KMS / GCP KMS / age keys; transparent `sops exec-env` integration
- **[age](https://github.com/FiloSottile/age)** — simple file encryption; combine with `sops` or use directly
- **[pass](https://www.passwordstore.org/)** — GPG-based password store; scriptable via `pass show verda/secrets`

Until migrated, ensure `./secrets` is in `.gitignore` (already configured), has mode 600, and is never committed or transmitted.
